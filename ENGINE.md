# Memory Framework Engine (v2)

> Agnostic kernel. It does not know which agent runs it, which methodology is in use, or which project is being worked on.
> It only provides primitive services that any combination of agent + methodology + project needs.

## 1. Boot Protocol

The agent orients itself in ≤30 seconds. Fixed load order (from general to specific):

1. **Engine** (this file) — primitive services and contract
2. **Methodology** (METHODOLOGY.yaml) — how this team works
3. **System index** (SYSTEM.yaml) — which documentation exists (index only, NOT content)
4. **Active work units** (via provider + cache) — names, states, owners
5. **Ready to work** — document and work unit contents loaded on demand

Universal entry point: `AGENTS.md`. Optional per-agent bridges (e.g. `CLAUDE.md` → redirect to `AGENTS.md`).

If METHODOLOGY.yaml does not exist or cannot be parsed, the agent informs the user which fields are missing. It does not assume implicit defaults.

## 2. Persistence Layer

Providers are content-agnostic. A provider does not know whether it stores a CURRENT_STATUS or an ARCHITECTURE — it only persists named artifacts.

| Operation | Description |
|-----------|-------------|
| `read(artifact_id)` | Read an artifact's contents |
| `write(artifact_id, content, mode)` | Write an artifact (replace/append) |
| `create(work_unit, artifact_name)` | Create a new artifact inside a work unit |
| `list(work_unit)` | List the artifacts in a work unit |
| `create_work_unit(name)` | Create a new work unit |
| `list_work_units(filter)` | List work units (filterable by state, owner) |
| `update_work_unit(id, state)` | Update a work unit's state |
| `read_system_doc(name)` | Read a system document (layer 3) |
| `write_system_doc(name, content)` | Write a system document |

**Provider cache**: auto-generated by the agent, gitignored, local to each machine. Warm start if it exists, cold start if not. It is updated incrementally as entities are created. Completed work units are removed from the cache.

**Two kinds of data in the cache — different behavior**:

| Section | Nature | Behavior |
|---------|--------|----------|
| Infrastructure, reference docs | **Stable** — IDs that do not change | Use the cache directly |
| Work unit list (projects) | **Dynamic** — new work units are added continuously | Cache = reference of known IDs, NOT a complete list |

**Discovery rule**: to list or discover active work units, always query the provider registry live. The cache resolves IDs for already-known work units; it never guarantees exhaustiveness. Using the cache as the complete project list is an anti-pattern that causes silent omissions.

## 3. Retrieval Protocol

Every work unit has an **entry point** — the artifact that lets you resume without asking.

Mandatory minimum fields for the entry point:
- `last_updated` — the agent detects staleness
- `next_step` — concrete action to resume

The agent reads the entry point and knows exactly where to continue.

## 4. Namespace & Registry

- **Work unit**: container for related artifacts (a project, a ticket, a spec)
- **Registry**: index of work units together with their state
- **State**: at least `active` and `completed` (the methodology may define more)
- **Ownership** (optional): for multi-user scenarios, via CONFIG.md (`current_user`)

## 5. Context Budget

- **Lazy loading** by default — do not load what is not needed
- **Layered principle** — load from general (engine) to specific (work unit)
- **Distillation** — compress session knowledge into durable artifacts
- **Minimum trigger**: explicit user request (always available, invariant)
- Never load multiple logs, inactive modules, or unrequested documents
- **Documentation-first exploration**: before exploring code (reading source files, searching with Glob/Grep, launching exploration agents), load all system documentation relevant to the area first — architecture pages, module context, API docs. Code exploration must happen *after* the available context is loaded, not before. This ensures the agent searches with intent rather than discovering blindly what the documentation already describes.

**Lazy loading with explicit trigger**: documents marked as `on_demand` in SYSTEM.yaml require a defined activation criterion. That criterion may live in:
- `METHODOLOGY.yaml` → `resource_hints` section (recommended)
- `SYSTEM.yaml` → `trigger` field on the document entry

Without a defined criterion, the agent **MUST ask the user** before accessing external sources directly (DB, APIs, etc.). The absence of a trigger is not permission to guess — it is a signal that configuration is missing.

## 6. Methodology Contract (METHODOLOGY.yaml)

The methodology registers itself with the engine through a YAML manifest.

**Required fields** (validated by the engine):
- `identity` (name, version), `work_unit` (label, entry_point, entry_point_fields with last_updated + next_step)
- `states.values` (at least 1 active + 1 completed), `distillation.always_update` (must include entry_point)

**Optional fields** (with defaults): `boot_context`, `artifacts.standard`, `distillation.triggers` (default: [user_request]), `distillation.targets_by_phase`

**Free zone** (`conventions`): extensible without limit. The engine ignores it but preserves it. The methodology defines its formatting, language, and naming conventions here, along with any team rules.

**Restriction**: `user_request` is always a distillation trigger (engine invariant).

## 7. Coexistence with native agent memories

The framework is **complementary**, not a replacement. It does not replace or compete with the agent's native memories (auto-memory, Memories, etc.).

**Priority rule**: on conflict, the framework prevails (explicit, versioned, human-curated).

## 8. Invariants

- Do not rely on conversation transcripts as memory
- Do not duplicate information across artifacts — reference it
- Distill before losing context
- The user can always request distillation
